数据库-入侵防范-数据库系统遵循最小安装的原则，仅安装需要的组件和应用程序，并通过升级等方式保持系统补丁及时

1. 确认数据库部署形态
2. 检查最小安装
3. 检查补丁更新机制
4. 形成证明材料和整改结论

一、检查目标和判定标准

先统一口径，不然后面容易答偏。

你这里要证明的不是：

- “H2 是否像 MySQL 一样单独安装”
- “虚拟机能否联网自动更新”

你要证明的是：

- 虚拟机没有安装不必要的数据库系统或组件
- 当前只使用 H2，且只启用业务所需能力
- 没有暴露数据库管理界面/管理端口
- 补丁更新虽然不能联网，但有离线更新机制，并且能执行、能留痕

最终结论通常写成：

- 数据库采用 Spring Boot 内嵌 H2 文件模式，不部署独立数据库服务
- 数据库仅包含业务必需组件，未安装无关数据库服务
- H2 Console 关闭，未启用 H2 TCP Server，未开放数据库管理端口
- 操作系统及应用补丁通过“联网构建机/堡垒机下载，离线上传到虚拟机”的方式更新

三、第一部分：确认数据库实际部署形态

这一步是为了证明：你不是装了一个完整数据库服务，而是应用内嵌 H2。

### 1. 查看应用配置文件

如果部署时配置在 /opt/deedm/application.yml，执行：

grep -nE 'datasource|jdbc:h2|driver-class-name|h2:|console:' /opt/deedm/application.yml

你要重点看这些内容：

- jdbc:h2:file:/opt/deedm/data/deedm
- driver-class-name: org.h2.Driver
- spring.h2.console.enabled: false

理想判定：

- 数据库 URL 指向本地文件
- 驱动是 H2
- H2 Console 已关闭

### 2. 查看数据库文件是否存在

ls -l /opt/deedm/data

理想情况会看到类似：

- deedm.mv.db

这说明数据库是本地文件库，而不是独立网络数据库。

### 3. 查看服务进程

ps -ef | grep -E 'java|h2' | grep -v grep

理想判定：

- 只有 Spring Boot 的 Java 进程
- 没有独立 h2 server 进程
- 没有 mysql、postgres 等进程

### 4. 查看 systemd 服务

systemctl status myapp --no-pager
systemctl list-unit-files | egrep -i 'mysql|mariadb|postgresql|mongod|redis|h2'

理想判定：

- 只有 myapp 这类业务服务
- 没有独立数据库服务单元

四、第二部分：检查“最小安装”是否满足

这一步是扫描项的核心。

## A. 检查操作系统层是否安装了不必要数据库软件

### 1. 列出已安装数据库相关软件包

openEuler 下常用：

rpm -qa | sort | egrep -i 'h2|mysql|mariadb|postgresql|mongo|redis|sqlite|oracle|db2'

如果还想看更完整一点：

dnf list installed | egrep -i 'h2|mysql|mariadb|postgresql|mongo|redis|sqlite|oracle|db2'

理想判定：

- 没有 MySQL、MariaDB、PostgreSQL、MongoDB、Redis 等数据库服务包
- 如果出现 sqlite-libs 这类系统基础库，不一定算问题，要区分“库”与“服务”
- H2 通常不会以 OS 包形式安装，因为它多半是打进 Jar 的 Java 依赖

### 2. 检查数据库服务是否启用

systemctl list-unit-files | egrep -i 'mysqld|mariadb|postgresql|mongod|redis'

如有命中，再看状态：

systemctl status <服务名> --no-pager

理想判定：

- 没有启用、没有运行无关数据库服务

## B. 检查网络暴露面

### 1. 查看监听端口

ss -lntp

如果只看数据库常见端口：

ss -lntp | egrep '(:3306|:5432|:27017|:6379|:1521|:1433|:9092|:8082)'

理想判定：

- 没有数据库管理端口监听
- 只看到 80/443 的 Nginx，以及 127.0.0.1:8080 的 Spring Boot 更合理
- 不应看到 H2 Console 常用端口或其他数据库端口

### 2. 检查防火墙暴露

firewall-cmd --list-all

理想判定：

- 只开放业务需要的 http/https
- 没有开放 3306、5432 等数据库端口

## C. 检查应用层是否真的只用了 H2

这是你这个场景特别关键的一步。因为即使 OS 没装 PostgreSQL，Spring Boot 的生产 Jar 里仍可能带了无关数据库驱
动。

### 1. 查看当前应用 Jar 中的数据库依赖

unzip -l /opt/deedm/deedm-backend-1.0.0.jar | egrep 'BOOT-INF/lib/(h2|postgresql|mysql|mariadb).*jar'

理想判定：

- 最好只看到 h2-xxx.jar
- 如果看到 postgresql-xxx.jar，说明生产包里带了未使用数据库驱动

### 2. 结合源码依赖检查

你当前源码里：

- 使用 H2：见 backend/pom.xml:50
- 同时保留了 PostgreSQL 运行时依赖：见 backend/pom.xml:56
- 同时保留了 devtools：见 backend/pom.xml:100

这两个在生产环境都建议处理：

- postgresql：如果生产只用 H2，建议删除
- spring-boot-devtools：建议不要进入生产包

### 3. 检查 H2 控制台是否关闭

grep -nA3 'h2:' /opt/deedm/application.yml

理想判定：

- enabled: false

### 4. 检查是否启用 H2 TCP Server

通常你的应用没有显式启用 H2 TCP Server，但还是建议确认：

ps -ef | grep -i h2 | grep -v grep
ss -lntp | egrep '(:9092|:8082)'

理想判定：

- 没有 H2 TCP/Web Server

五、第三部分：检查“补丁及时更新”如何证明

由于虚拟机不能连外网，这里不要试图证明“在线自动更新”，而要证明：

- 有明确的离线更新流程
- 可以定期从外部可信环境获取补丁
- 可以通过堡垒机上传并实施更新
- 有更新记录和版本对比记录

这在等保/审计里通常是可接受的。

## A. 检查操作系统补丁状态

### 1. 查看系统版本

cat /etc/os-release
uname -r

### 2. 查看近期更新历史

openEuler 可以尝试：

dnf history

如果想看最近更新的包：

rpm -qa --last | head -n 50

理想判定：

- 能看到系统更新记录
- 至少能证明系统维护过，不是长期无更新

如果完全没有更新记录，也不一定立即不合格，但需要补充制度说明。

## B. 检查 Java 和 Nginx 版本

java -version
nginx -v
rpm -qa | egrep 'java-17-openjdk|nginx'

这属于基础软件版本取证。

## C. 检查应用依赖版本

### 1. 本地源码确认 Spring Boot 版本

grep -n 'spring-boot-starter-parent' -A3 /home/admin-unix/deedm_alpha/backend/pom.xml

当前源码中是 3.2.0，见 backend/pom.xml:8。

### 2. 确认 H2 实际打包版本

在构建机或本地源码目录执行：

cd /home/admin-unix/deedm_alpha/backend
mvn dependency:tree | grep com.h2database:h2

或者打包后：

unzip -l target/deedm-backend-1.0.0.jar | grep 'BOOT-INF/lib/h2-'

### 3. 保留版本对比记录

建议形成一张表：

- 当前 Spring Boot 版本
- 当前 H2 版本
- 上次更新时间
- 本次检查时间
- 是否存在待升级项

六、第四部分：离线补丁更新的标准执行流程

这是你最需要的一块。下面给一套适合你环境的标准流程。

## A. 应用依赖补丁更新流程

适用于 H2、Spring Boot、Jasypt、Springdoc 等 Java 依赖升级。

### 步骤 1：在联网构建机确认升级目标

在可联网环境完成：

- 确认 H2 是否有新版本
- 确认 Spring Boot 是否有安全更新
- 评估兼容性

### 步骤 2：修改源码依赖

如果继续跟 Spring Boot BOM 走，就升级 Spring Boot 父版本。
如果只想单独锁定 H2，也可以在 pom.xml 显式指定 H2 版本。

生产建议同时做两项整改：

- 删除 postgresql
- 删除 spring-boot-devtools

### 步骤 3：在构建机打包

cd /path/to/deedm_alpha/backend
mvn clean -DskipTests package

### 步骤 4：核对打包产物中的依赖

unzip -l target/deedm-backend-1.0.0.jar | egrep 'BOOT-INF/lib/(h2|postgresql|spring-boot-devtools).*jar'

理想结果：

- 有 h2
- 没有 postgresql
- 没有 spring-boot-devtools

### 步骤 5：计算校验值

sha256sum target/deedm-backend-1.0.0.jar > target/deedm-backend-1.0.0.jar.sha256

### 步骤 6：通过堡垒机上传到虚拟机

上传到例如：

- /opt/deedm/upgrade/
或
- 临时上传到用户目录，再 sudo mv

### 步骤 7：虚拟机实施更新

先备份：

sudo systemctl stop myapp
sudo cp /opt/deedm/deedm-backend-1.0.0.jar /opt/deedm/backup/deedm-backend-1.0.0.jar.$(date +%F_%H%M%S)

校验新包：

cd /opt/deedm/upgrade
sha256sum -c deedm-backend-1.0.0.jar.sha256

替换：

sudo cp /opt/deedm/upgrade/deedm-backend-1.0.0.jar /opt/deedm/deedm-backend-1.0.0.jar
sudo chown springapp:springapp /opt/deedm/deedm-backend-1.0.0.jar

启动并检查：

sudo systemctl start myapp
sudo systemctl status myapp --no-pager
sudo journalctl -u myapp -n 100 --no-pager

再次核对监听情况：

ss -lntp

### 步骤 8：记录更新结果

至少记录：

- 更新日期
- 更新前版本
- 更新后版本
- 上传人/操作人
- 校验值
- 服务启动结果
- 回滚包位置

———

## B. 操作系统补丁离线更新流程

适用于 openEuler 系统补丁。

### 步骤 1：联网环境准备 RPM 包

在联网的 openEuler 同版本环境或仓库机上下载补丁包。
如果能做离线仓库更好；如果不能，至少下载本次更新涉及的 RPM。

### 步骤 2：通过堡垒机上传

上传到虚拟机如：

/opt/offline_rpms/

### 步骤 3：核对包完整性

ls -lh /opt/offline_rpms
rpm -K /opt/offline_rpms/*.rpm

### 步骤 4：执行离线更新

sudo dnf update /opt/offline_rpms/*.rpm

如果是本地仓库方式，按本地 repo 配置执行。

### 步骤 5：重启并验证

sudo reboot

重启后检查：

uname -r
cat /etc/os-release
rpm -qa --last | head -n 30

### 步骤 6：记录

记录：

- 更新时间
- 更新包清单
- 更新前后内核/关键组件版本
- 是否重启
- 验证结果

八、检查结果怎么判定

## 通过的标准

满足以下多数条件即可判定基本通过：

- /opt/deedm/application.yml 显示使用 jdbc:h2:file:...
- H2 Console 为 false
- 无独立数据库服务
- 无数据库管理端口对外监听
- 虚拟机未安装无关数据库服务包
- 有操作系统和应用的离线更新流程
- 有更新历史或制度性记录

## 需要整改的点

结合你当前项目，最可能的整改项有两个：

1. 生产包中仍包含 postgresql 驱动
    原因：源码中还有 PostgreSQL 依赖，见 backend/pom.xml:56
2. 生产包中可能包含 spring-boot-devtools
    原因：源码中保留了 devtools，见 backend/pom.xml:100

这两项建议在下次发版前清理掉。否则如果扫描较严格，会被认为“组件未按最小化原则部署”。
