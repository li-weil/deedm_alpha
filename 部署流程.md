### 创建spingapp用户管理sping boot应用
在 Linux 中，每个进程都由某个用户身份运行。这个用户决定了进程对文件系统、网络、系统资源的访问权限。
当你创建一个专用用户 springapp 并用它运行 Spring Boot 应用时：
该用户只拥有对 /opt/deedm 的访问权限（你通过 chown springapp:springapp /opt/deedm 设置的）
它无法访问其他用户的文件或系统敏感目录（如 /root），从而降低了安全风险
```bash
sudo useradd -r -s /bin/false springapp                                           #添加springapp用户
sudo mkdir -p /opt/deedm                              
sudo chown springapp:springapp /opt/deedm                               #赋予 /opt/myapp 的访问权限
```

/etc/systemd/system/myapp.service

systemd 配置中指定了该用户，并显式导入 `/opt/deedm/.env` 中的 Jasypt 密钥：
```ini

[Unit]
Description=My Spring Boot App
After=network.target

[Service]
User=springapp
EnvironmentFile=/opt/deedm/.env
WorkingDirectory=/opt/deedm
ExecStart=/usr/bin/java $JAVA_OPTS -jar /opt/deedm/deedm-backend-1.0.0.jar --spring.config.location=/opt/deedm/application.yml --jasypt.encryptor.password=${JASYPT_KEY}
SuccessExitStatus=143
Restart=on-failure
RestartSec=10
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target

```
**关键说明** 使用jasypt加密，秘钥在opt/deedm/.env，权限设置为root所有者
```bash
sudo chown root:root /opt/deedm/.env
sudo chmod 600 /opt/deedm/.env
```
`/opt/deedm/.env` 内容示例：
```bash
JASYPT_KEY=your-jasypt-master-password
AUDIT_INTEGRITY_SECRET=replace-with-a-long-random-secret
```

新增审计保护说明：
1. `AUDIT_INTEGRITY_SECRET` 用于生成 `audit_log.record_hash`，建议使用至少 32 字节随机值，并与数据库文件分开保管。
2. 审计表 `audit_log` 默认按“只追加”方式保护，普通 `UPDATE` / `DELETE` 会被数据库触发器拒绝。
3. 审计保留期清理仍允许执行，但只能通过应用内受控维护逻辑触发，普通业务路径不能直接删改审计记录。

当前仓库里的 `spring.datasource.password` 密文是按以下 Jasypt 参数生成的，运行时必须保持一致，否则 Spring 会在绑定 `spring.datasource.password` 时直接启动失败：
```yml
jasypt:
  encryptor:
    algorithm: PBEWithMD5AndDES
    iv-generator-classname: org.jasypt.iv.NoIvGenerator
    string-output-type: base64
```
如果以后重新生成 `ENC(...)`，要么继续使用这组参数，要么连同这里的参数一起更新。

建议把 `~/encrypt/src/main/java/com/example/CipherGeneratorApp.java` 视为“参数参考”，不是“最终生产流程”。它现在能说明当前密文体系基于 `PBEWithMD5AndDES`，但它默认会随机生成“明文密码”和“主密钥”，这不适合实际部署。更稳妥的发布流程应当固定为：
1. 你自己先决定数据库明文密码，例如 H2 的 `sa` 密码。
2. 你自己先决定一个长期保存于 `/opt/deedm/.env` 的 `JASYPT_KEY`。
3. 本地加密程序只负责做一件事：用这组固定输入生成 `ENC(...)`。
4. 服务器上只保存 `ENC(...)` 和 `JASYPT_KEY`，不要再现场生成或临时改密钥。
5. 不要为了追求“可复现”而在生产加密时传 `seed` 固定盐；正常随机盐即可，因为盐会编码进密文里，Jasypt 解密不依赖外部保存盐值。


### 文件上传
更改目录所有者： 将目录的所有权交给你的当前登录用户，这样你就能直接上传文件了，记得传完之后要改回springapp：
```bash
sudo chown -R u01:u01 /opt/deedm
sudo chown -R springapp:springapp /opt/deedm
```
把 demo-0.0.1-SNAPSHOT.jar 上传到 /opt/deedm/

上传完成后，务必再次确认运行用户 `springapp` 对数据库文件目录有写权限，否则 H2 可能以只读方式打开文件并触发 `AUTO_SERVER/readOnly` 类错误：
```bash
sudo chown -R springapp:springapp /opt/deedm
sudo chmod 750 /opt/deedm
sudo chmod 750 /opt/deedm/data
```

```bash
sudo mkdir -p /var/www/deedm
sudo chown -R u01:u01 /var/www/deedm
```

把 dist.zip上传到 /var/www/deedm
```bash
cd /var/www/deedm && unzip dist.zip

sudo chown -R nginx:nginx /var/www/deedm
#赋予nginx权限(这是openeuler系统，nginx对应用户名就是nginx，ubuntu是www-data)
```

### 后端配置
/opt/deedm/application.yml

```yml
server:
  address: 127.0.0.1
  port: 8080
  servlet:
    context-path: /api

spring:
  application:
    name: deedm-backend

  # Database configuration
  datasource:
  # 单机部署使用 H2 文件模式，数据库文件位于 /opt/deedm/data/deedm.mv.db
  # 当前架构是“多个客户端 -> 一个 Spring Boot 进程 -> 一个 H2 文件库”
  # 因此不需要 AUTO_SERVER=TRUE；它主要用于多个独立进程同时打开同一个 H2 文件
    url: jdbc:h2:file:/opt/deedm/data/deedm
    driver-class-name: org.h2.Driver
    username: sa
    password: "ENC(GC1RBSC9zFRC5Opq4bXAVRRN5P821xdzFpUcQEHFlhk=)"

  # JPA configuration
  jpa:
    database-platform: org.hibernate.dialect.H2Dialect
    open-in-view: false
    hibernate:
      ddl-auto: update
    show-sql: false
    properties:
      hibernate:
        format_sql: true

  # H2 Console configuration,close console
  h2:
    console:
      enabled: false
      path: /h2-console

jasypt:
  encryptor:
    algorithm: PBEWithMD5AndDES
    iv-generator-classname: org.jasypt.iv.NoIvGenerator
    string-output-type: base64

# Logging configuration,DEBUG->INFO
logging:
  file:
    name: ./logs/deedm-backend.log
  level:
    com.deedm: INFO
    org.springframework.web: INFO
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} - %msg%n"

audit:
  enabled: true
  integrity:
    secret: "${AUDIT_INTEGRITY_SECRET:}"
  abuse:
    enabled: true
    threshold-per-minute: 120
  retention:
    days: 180
    cron: "0 30 2 * * *"

# Application configuration
deedm:
  # CORS configuration
  cors:
    allowed-origins: "127.0.0.1"
    allowed-methods: GET,POST,PUT,DELETE,OPTIONS
    allowed-headers: "*"

  # File upload configuration
  upload:
    max-size: 10MB
    allowed-types: .txt,.json

  # Cache configuration
  cache:
    ttl: 3600 # seconds

# OpenAPI configuration
springdoc:
  api-docs:
    path: /api-docs
  swagger-ui:
    path: /swagger-ui.html

```

说明：
1. 如果你确实需要在 Spring Boot 运行期间，再从另一个独立进程同时打开同一个 H2 文件库（例如长期使用外部 H2 Shell/H2 Console 直连同一文件），再考虑恢复 `AUTO_SERVER=TRUE`。
2. 仅仅是“不同 IP 的浏览器同时访问网站”，不需要 `AUTO_SERVER=TRUE`，因为最终只有一个 Spring Boot 进程在访问 H2。
3. H2 Shell 不会解密 `ENC(...)`，直连数据库时仍必须使用数据库明文密码。
4. 若你继续沿用 `CipherGeneratorApp` 生成密文，应用侧配置必须与它保持一致；当前仓库已经按它的 `PBEWithMD5AndDES` 体系固定好了 `jasypt.encryptor.*`。

### nignx配置
openEuler系统，/etc/nginx/nginx.conf为默认配置，/etc/nginx/conf.d/myapp.conf为个性配置

需注释nginx.conf中的server才能让个性配置起作用
/etc/nginx/conf.d/myapp.conf
```conf
server {
    listen 80;
    server_name _;

    root /var/www/deedm/dist;
    index index.html;

    location ^~ /.well-known/acme-challenge/ {
        default_type "text/plain";
        root /var/www/deedm;   # webroot 的上一级
        try_files $uri =404;
        # 若有 auth_basic 或 IP 白名单，不能在这里启用
    }


    # 静态文件优先
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 反代
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

}
```
**注意** root要写到index.html所在目录

```bash

sudo nginx -t && sudo systemctl reload nginx

```

#### 2. 重要用户行为检测（写接口/文件删除）
触发写接口（示例：清空白名单）：
```bash
curl -X POST "http://127.0.0.1/api/cleanup/data" \
  -H "Content-Type: application/json" \
  -H "X-Visitor-Id: scan-user-a" \
  -d '{"filenames":["AST_test.png","Relation_test.png"]}'
```

检查日志证据：
```bash
grep "AUDIT" /opt/deedm/logs/deedm-backend.log | tail -n 30
```

检查数据库证据：
```sql
SELECT id, created_at, event_type, severity, action, request_path, http_method,
       visitor_id, client_ip, status_code, duration_ms, success, error_message
FROM audit_log
WHERE event_type IN ('API_WRITE', 'FILE_DELETE')
ORDER BY id DESC
LIMIT 30;
```

#### 3. 高风险读取拒绝事件检测（非法/异常文件访问）
触发高风险读取失败（应返回 4xx）：
```bash
curl -i "http://127.0.0.1/api/formula-syntax/ast-image/AST_deadbeef.png" \
  -H "X-Visitor-Id: scan-read-a"
```

核对审计事件：
```sql
SELECT id, created_at, event_type, severity, request_path, status_code, visitor_id, client_ip
FROM audit_log
WHERE event_type = 'FILE_READ_DENY'
ORDER BY id DESC
LIMIT 20;
```

#### 4. 重要系统命令使用检测（Graphviz dot）
通过页面或接口触发一次图片生成功能（AST、哈斯图、图可视化任一均可）。

核对 `CMD_EXEC` 事件（命令执行成功/失败都应留痕）：
```sql
SELECT id, created_at, event_type, severity, action, request_path, duration_ms, success, error_message
FROM audit_log
WHERE event_type = 'CMD_EXEC'
ORDER BY id DESC
LIMIT 20;
```

#### 5. 系统资源异常使用检测（仅审计不拦截）
模拟高频访问高风险接口：
```bash
for i in $(seq 1 140); do
  curl -s -o /dev/null -H "X-Visitor-Id: scan-abuse-a" \
  "http://127.0.0.1/api/formula-syntax/ast-image/AST_deadbeef.png"
done
```

核对异常事件：
```sql
SELECT id, created_at, event_type, severity, request_path, resource_key, threshold, error_message
FROM audit_log
WHERE event_type = 'RESOURCE_ABUSE'
ORDER BY id DESC
LIMIT 20;
```

说明：当前策略为“记录不拦截”，即即使出现 `RESOURCE_ABUSE`，接口不主动返回 429。

#### 6. 审计保留策略检测（180 天）
检查配置已生效并确认有维护类事件：
```sql
SELECT id, created_at, event_type, threshold, success, error_message
FROM audit_log
WHERE event_type = 'AUDIT_MAINTENANCE'
ORDER BY id DESC
LIMIT 20;
```

#### 7. Controller 调试信息降噪检查
生产日志只保留结构化审计事件，不应出现大段 Controller `System.out` 调试输出。
```bash
grep -E "Controller" /opt/deedm/logs/deedm-backend.log | tail -n 20
```

#### 8. 双访问者清空隔离验证
1. A、B 两个浏览器分别生成图片。  
2. A 点击“清空”后，仅删除 A 本次页面传入白名单文件。  
3. B 的图片仍可访问。  

#### 9. 扫描通过判定标准
1. 文件日志与 H2 `audit_log` 均可检索到事件。  
2. 至少可检索到事件类型：`API_WRITE`、`FILE_DELETE`、`FILE_READ_DENY`、`CMD_EXEC`、`RESOURCE_ABUSE`。  
3. 审计记录包含访问者身份（`visitor_id`）、来源地址（`client_ip`）、接口路径、结果状态和时间。  
4. 日志中无大量 Controller 原始调试打印噪音。
