# 数据有效性检验整改方案

## 1. 整改目标

针对安全扫描项“应用安全-软件容错-应提供数据有效性检验功能，保证通过人机接口输入或通过通信接口输入的数据格式或长度符合系统设定要求”，本次整改目标如下：

1. 为前后端输入建立明确的长度、格式、范围约束。
2. 在后端建立独立于前端的统一校验能力，避免仅依赖页面提示。
3. 对超长请求、非法字符、异常 JSON 等输入返回明确的 `4xx` 响应。
4. 保留现有业务校验逻辑，同时补足统一入口校验与异常处理。

## 2. 适用范围

本方案适用于当前离散数学教学 Web 应用的以下输入面：

1. Vue 页面中的文本输入框和文本域。
2. Spring Boot 后端 `POST /api/**` 的 JSON 请求体。
3. 图片读取类接口中的路径参数 `filename`。

不包含文件上传场景，因为系统当前不提供文件上传能力。

## 3. 现状问题

经检查，当前系统存在以下问题：

1. 前端多数输入框仅有占位提示，没有统一 `maxlength` 或长度提示。
2. 后端大量接口直接接收 `@RequestBody`，但缺少统一的请求体校验入口。
3. 请求模型类普遍未设置统一长度约束。
4. 服务层存在部分业务校验，但分散在各模块中，覆盖不一致。
5. 部署文档中未明确 Spring Boot 与 Nginx 的请求体大小限制。

## 4. 整改策略

### 4.1 后端统一校验

在 Spring Boot 中新增统一输入校验层，覆盖全部 JSON 请求体：

1. 对请求体总字节数进行限制，超过阈值直接返回 `413 Payload Too Large`。
2. 对请求体中的字符串字段进行统一校验：
   - 长度不得超过系统设定上限。
   - 不允许包含不可见控制字符。
   - 对文件名类字段采用更严格的长度限制。
3. 对集合、数组、Map 的元素数量进行限制，避免异常膨胀输入。
4. 对嵌套对象递归校验，限制最大递归深度。
5. 通过全局异常处理将异常统一返回为 `400/413`，避免暴露栈信息。

### 4.2 保留业务层格式校验

各模块现有的业务校验逻辑继续保留，例如：

1. 组合数接口对 `n`、`m` 的范围限制。
2. 关系、图、公式等模块对表达式语义合法性的判断。
3. 图片读取接口对 `filename` 的正则校验。

统一校验负责“长度、非法字符、结构规模”，业务校验负责“语义与规则”。

### 4.3 前端输入约束

建议在后续前端版本中同步补充：

1. 为主要输入框添加 `maxlength`。
2. 提交前增加本地长度校验与提示。
3. 对数字输入控件补充取值范围说明。

说明：前端校验用于提升用户体验，不作为唯一安全边界，最终以后端校验结果为准。

### 4.4 部署层限制

生产环境建议同时配置：

1. Nginx `client_max_body_size`。
2. Spring Boot 请求头与请求体限制参数。
3. 审计日志对 `400/413` 拒绝事件留痕。

### 5.3 本次代码改动的实现逻辑

本次实际落地的代码逻辑如下：

1. 对 `POST/PUT/PATCH` 且 `Content-Type` 为 `application/json` 的请求统一进行请求体大小限制。
2. 当请求体字节数超过阈值时，由过滤器直接返回 `413 Payload Too Large`，返回 JSON 格式错误信息。
3. 当请求体进入控制器前，对对象中的字符串、集合、数组、Map、嵌套对象执行统一递归校验。
4. 对普通字符串字段采用默认长度上限。
5. 对 `formula`、`expression`、`relation`、`function`、`nodes`、`edges`、`set` 等复杂输入字段采用扩展长度上限。
6. 对 `filename` 字段采用更严格长度限制。
7. 对字符串中的非法控制字符进行拦截，允许的控制字符仅保留换行、回车、制表符。
8. 对集合、数组、Map 的元素数量进行限制，避免构造异常膨胀请求。
9. 对对象嵌套层级进行限制，避免异常深层输入。
10. 原有业务服务中的语义校验逻辑继续保留，不与统一校验冲突。

## 6. 校验基线建议

建议先采用以下默认基线：

1. 单次 JSON 请求体最大 `64KB`。
2. 普通字符串字段默认最大长度 `1024`。
3. 公式、关系、集合、表达式、图节点边描述等复杂文本最大长度 `4096`。
4. 文件名类字段最大长度 `128`。
5. 集合、数组、Map 最大元素数 `200`。
6. 嵌套对象最大深度 `8`。

后续可根据真实业务再做细化。
