# 数据库-安全审计-审计记录保护整改报告

## 1. 背景

本次整改目标对应安全审计要求：

> 应保护审计记录，避免受到未预期的删除、修改或覆盖等。

当前项目部署架构为：

- 前端：Vue
- 后端：Spring Boot
- 反向代理：Nginx
- 数据库：H2 文件库
- 部署方式：Nginx + Spring Boot 部署在虚拟机，当前本机仅保存代码与部署文档

现有审计记录主要写入 H2 数据库表 `audit_log`，同时输出部分结构化审计日志到应用日志文件。

## 2. 整改前风险

整改前，`audit_log` 虽然承担审计记录功能，但本质仍是普通业务表，存在以下问题：

1. 审计表缺少“只追加”约束，理论上可被 `UPDATE`、`DELETE`。
2. 审计记录没有内建完整性校验字段，离线篡改后难以及时发现。
3. 保留期清理逻辑虽然存在，但缺少“受控维护上下文”和“普通路径禁止删改”的隔离表达。
4. 部署文档中没有单独要求配置审计完整性密钥。

### 3.2 新增审计记录完整性计算服务

文件：

## 5. 安全效果说明

本次整改完成后，安全效果如下：

1. 审计记录不再是普通可改可删表，默认进入“只追加”状态。
2. 普通业务路径不能直接修改或删除审计记录。
3. 合法的保留期清理仍保留，但必须经过受控维护逻辑。
4. 每条审计记录均带有摘要校验值，便于后续做离线篡改核验。
5. 旧记录在升级后也会补齐摘要字段，避免保护断层。

## 6. 仍然存在的边界

需要明确，本次整改是在“单机 H2 文件库”架构内做增强，不能等同于外部独立审计系统。

当前仍有以下边界：

1. 如果攻击者已获得虚拟机高权限，并能同时读取数据库文件与完整性密钥，则仍可能离线篡改并重算摘要。
2. H2 文件库本身不适合承担高强度防篡改审计库职责。
3. 本次实现可以显著降低“应用误操作、普通路径误删改、低门槛直连修改”的风险，但不能替代独立审计平台。

## 7. 建议在虚拟机上执行的手工操作

### 7.1 配置完整性密钥

编辑 `/opt/deedm/.env`，新增：

```bash
AUDIT_INTEGRITY_SECRET=replace-with-a-long-random-secret
```

建议要求：

1. 长度至少 32 字节。
2. 使用随机值，不与 `JASYPT_KEY` 复用。
3. 继续保持：

```bash
sudo chown root:root /opt/deedm/.env
sudo chmod 600 /opt/deedm/.env
```

### 7.2 部署新版本并重启服务

```bash
sudo systemctl daemon-reload
sudo systemctl restart myapp
sudo journalctl -u myapp -n 100 -f
```

### 7.3 收紧数据库文件目录权限

建议确认：

```bash
sudo chown -R springapp:springapp /opt/deedm
sudo chmod 750 /opt/deedm
sudo chmod 750 /opt/deedm/data
```

原则：

1. `springapp` 拥有写权限。
2. 普通登录用户不应拥有 `/opt/deedm/data` 写权限。
3. 非必要情况下，不要长期把 `/opt/deedm` 改回普通用户可写。

### 7.4 做现场验证

建议验证以下项目：

1. 正常访问业务接口后，`audit_log` 能继续新增记录。
2. 手工执行 `UPDATE audit_log ...` 应失败。
3. 手工执行 `DELETE FROM audit_log ...` 应失败。
4. 定时保留期清理到期后仍能删除过期记录。
5. 新记录已具备 `record_hash`。

### 7.5 建立备份与归档

建议对以下对象建立 root 控制的定期备份：

1. `/opt/deedm/data/deedm.mv.db`
2. `/opt/deedm/logs/deedm-backend.log`
3. `/opt/deedm/application.yml`

目的：

- 防止数据库文件或日志文件被删除后无法追溯。
- 为离线校验和恢复提供依据。


#### 7.5.1 先创建 root 控制的备份目录：
```bash
  sudo mkdir -p /var/backups/deedm/log
  sudo chown -R root:root /var/backups/deedm
  sudo chmod -R 700 /var/backups/deedm
  sudo mkdir -p /var/backups/deedm/db
```

#### 7.5.2 手工执行一次数据库和日志备份（保持时间戳一致）：
```bash 
  TS=$(date +%F_%H%M%S)
  sudo cp /opt/deedm/data/deedm.mv.db /var/backups/deedm/db/deedm.mv.db.$TS
  sudo gzip /var/backups/deedm/db/deedm.mv.db.$TS

  sudo cp /opt/deedm/logs/deedm-backend.log /var/backups/deedm/log/deedm-backend.log.$TS
  sudo gzip /var/backups/deedm/log/deedm-backend.log.$TS
```
#### 7.5.3 加一个简单的保留清理，保留 30 天：
```bash
  sudo find /var/backups/deedm/db -type f -name '*.gz' -mtime +30 -delete
  sudo find /var/backups/deedm/log -type f -name '*.gz' -mtime +30 -delete
```


#### 7.5.4 更新/usr/local/sbin/deedm-backup.sh：
```bash
  #!/usr/bin/env bash
  set -euo pipefail

  BACKUP_ROOT="/var/backups/deedm"
  DB_SRC="/opt/deedm/data/deedm.mv.db"
  LOG_SRC="/opt/deedm/logs/deedm-backend.log"

  DB_DIR="$BACKUP_ROOT/db"
  LOG_DIR="$BACKUP_ROOT/log"
  TS="$(date +%F_%H%M%S)"

  mkdir -p "$DB_DIR" "$LOG_DIR"
  chown root:root "$BACKUP_ROOT" "$DB_DIR" "$LOG_DIR"
  chmod 700 "$BACKUP_ROOT" "$DB_DIR" "$LOG_DIR"

  if [ -f "$DB_SRC" ]; then
    cp "$DB_SRC" "$DB_DIR/deedm.mv.db.$TS"
    gzip -f "$DB_DIR/deedm.mv.db.$TS"
  fi

  if [ -f "$LOG_SRC" ]; then
    cp "$LOG_SRC" "$LOG_DIR/deedm-backend.log.$TS"
    gzip -f "$LOG_DIR/deedm-backend.log.$TS"
  fi

  find "$DB_DIR" -type f -name '*.gz' -mtime +30 -delete
  find "$LOG_DIR" -type f -name '*.gz' -mtime +30 -delete
```

#### 7.5.4 给权限
```bash
  sudo chown root:root /usr/local/sbin/deedm-backup.sh
  sudo chmod 700 /usr/local/sbin/deedm-backup.sh
```
#### 7.5.5 先手工试跑一次
```bash
  sudo /usr/local/sbin/deedm-backup.sh
  sudo ls -l /var/backups/deedm/db
  sudo ls -l /var/backups/deedm/log
```
#### 7.5.6 然后加到 root 的 cron，每天凌晨 03:10 执行：
```bash
  sudo crontab -e
```
  加入这一行：
```
  10 3 * * * /usr/local/sbin/deedm-backup.sh >/var/log/deedm-backup.log 2>&1
```

## 9. 结论

本次整改已在代码层完成以下关键提升：

1. 为审计表建立默认只追加保护。
2. 为审计记录增加完整性摘要。
3. 将保留期删除显式限定为受控维护行为。
